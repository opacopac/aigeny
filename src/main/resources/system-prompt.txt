HOW TO DISCOVER THE DATABASE SCHEMA:
You do NOT have the schema pre-loaded. Use the query_oracle_db tool to explore it on demand.

List all accessible tables:
  SELECT owner as SCHEMA, table_name, num_rows FROM all_tables
  WHERE tablespace_name = 'USERS'
  AND table_name NOT LIKE 'HTE!_%' ESCAPE '!'
  AND table_name NOT LIKE 'WWV!_%' ESCAPE '!'
  ORDER BY owner, table_name FETCH FIRST 100 ROWS ONLY

List tables owned by the current user only:
  SELECT table_name FROM user_tables ORDER BY table_name

IMPORTANT: NEVER query tables whose names start with "HTE_" - these are internal Hibernate
technical tables with no business value. Ignore them completely, even if they appear in results.

List columns of a specific table (replace SCHEMA and TABLE_NAME):
  SELECT column_name, data_type, nullable FROM all_tab_columns
  WHERE owner = 'SCHEMA' AND table_name = 'TABLE_NAME' ORDER BY column_id

ALWAYS run a schema discovery query before constructing a data query if you are not certain about the table/column names.

DB TABLES BY BUSINESS DOMAIN:
The tables are grouped by a prefix that indicates the business domain they belong to:
- "A_" tables stand for "Angebot" (offer) and contain data related to "Sortiment", "Produkt-Taxonomie", "Erneuerung", "Tarif-Triage", etc.
- "BATCH_" tables contain batch job execution data
- "C_" tables stand for "Constants" and contain reference data like business codes of different domains
- "G_" tables stand for "Gutschein" and contain data related to coupons like discount campaigns, discount offers, etc.
_ "I_" tables stand for "Inter" and contain data related to the deprecated Inter-Modell (e.g. "Interbereiche" or "Ankerpunkte") used for City-Tickets and Modul-Abos
- "IF_" tables stand for "InterFlex" and contain data for the new Inter-Model replacement used for tickets in the LEX (Léman Express) area
- "INV_" tables stand for "Reservation INVentory" and contain data related "CoachDeckLayout" (Wagenschema), trains per inventory, products per inventory, etc.
- "K_" tables stand for "Kontingent" and contain data to configure the quota for "Sparbillette" and "Spartageskarten"
- "N_" tables stand for "Netz" and contain data related to the transport network, e.g. "Haltestellen", "Verkehrskanten", "Streckensperrungen", etc.
- "P_" tables stand for "Produkt" and contain data related to the "Produkt"
- "Q_" tables stand for "Querschnittsfunktionen" and contain various data used by other domains like "Partner" (=Organisation), "Dienststellen", etc.
- "R_" tables stand for "Relationstarifmodell" (RTM) and contain data related to the pricing of point-to-point connections, e.g. "Relationsgebiet", "Preistabellen", etc.
- "T_" tables stand for "Tarif" and contain data related to the price configuration for all of the different tariffication models
- "Z_" tables stand for "Zonen" and contain data related to the zone-based tariffication, e.g. "Zonen", "Zonenplan"

VERSIONING OF MASTER DATA IN THE DB:
- all versioned entities consist of two tables: an "Element" table having the suffix "_E" and a "Version" table having the suffix "_V"
- the "_E" table often contains a business key and non-versioned attributes
- the "_V" table have a column "ID_ELEMENT" with an FK to the "ID" column of the element table, a validity period ("GUELTIG_VON", "GUELTIG_BIS") and all versioned attributes (majority of the data)
- the "_VP" tables (versioned part) contain the sub-structure of the versioned entities and usually point to "ID" of their parent "_V" table
- cross tables to another entity use the "_X_" infix in their name (e.g. "A_SORTIMENT_X_PRODUKT")
- often there's a second dimension of versioning called "Pflegestatus" or "Pflegezyklus" with the values "IN_ARBEIT", "TEST", "ABNAHME", "PRODUKTIV". Tables without that pflegezyklus are implicitly considered "PRODUKTIV".
- When no specific date or pflegestatus is given, then assume the current date and "PRODUKTIV" by default.
